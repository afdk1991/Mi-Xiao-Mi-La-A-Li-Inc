# 数据库设计方案

## 1. 数据库选型

| 数据库类型 | 用途 | 版本 |
|-----------|------|------|
| MySQL | 关系型数据库，存储结构化数据 | 8.0+ |
| MongoDB | 文档数据库，存储半结构化/非结构化数据 | 7.0+ |
| Redis | 内存数据库，缓存热点数据 | 7.0+ |
| Kafka | 消息队列，处理异步消息 | 3.0+ |

## 4. Redis 缓存设计

### 4.1 缓存键设计

| 缓存键 | 类型 | 过期时间 | 用途 |
|--------|------|----------|------|
| `user:{user_id}` | Hash | 24小时 | 用户信息缓存 |
| `order:{order_id}` | Hash | 1小时 | 订单信息缓存 |
| `product:{product_id}` | Hash | 1小时 | 商品信息缓存 |
| `merchant:{merchant_id}` | Hash | 1小时 | 商家信息缓存 |
| `token:{user_id}` | String | 15分钟 | 访问令牌 |
| `refresh_token:{user_id}` | String | 7天 | 刷新令牌 |
| `captcha:{phone}` | String | 5分钟 | 验证码 |
| `rate_limit:{ip}` | String | 1分钟 | IP限流 |
| `rate_limit:{user_id}` | String | 1分钟 | 用户限流 |
| `hot_products` | List | 10分钟 | 热门商品 |

### 4.2 缓存策略

- **Cache Aside**：读缓存→缓存未命中→读数据库→写缓存
- **延迟双删**：写数据库→删缓存→等待→删缓存
- **缓存预热**：系统启动时加载热点数据到缓存
- **缓存降级**：缓存不可用时，直接访问数据库

## 6. 数据同步方案

### 6.1 MySQL 与 Redis 同步

- **Cache Aside**：应用程序负责缓存的更新
- **Canal**：基于 MySQL Binlog 的增量同步

### 6.2 MySQL 与 MongoDB 同步

- **Debezium**：捕获 MySQL 变更并发送到 Kafka
- **Kafka Connect**：消费 Kafka 消息并写入 MongoDB

### 6.3 跨服务数据同步

- **事件驱动**：通过 Kafka 消息实现服务间数据同步
- **定时任务**：定期同步数据，确保数据一致性

## 7. 数据安全

### 7.1 数据加密

- **传输加密**：使用 HTTPS 传输数据
- **存储加密**：敏感数据（如密码、身份证号）加密存储
- **加密算法**：使用 AES-256 加密算法

### 7.2 数据备份

- **MySQL**：每日全量备份，每小时增量备份
- **MongoDB**：每日全量备份
- **Redis**：开启持久化（RDB + AOF）
- **备份存储**：异地存储备份数据

### 7.3 数据恢复

- **MySQL**：基于备份文件恢复
- **MongoDB**：基于备份文件恢复
- **Redis**：基于 RDB/AOF 文件恢复

## 8. 性能优化

### 8.1 数据库优化

- **索引优化**：合理创建和使用索引
- **查询优化**：优化 SQL 查询语句
- **分库分表**：数据量大时进行分库分表
- **读写分离**：主库写，从库读

### 8.2 缓存优化

- **多级缓存**：本地缓存 + 分布式缓存
- **缓存策略**：合理设置缓存过期时间
- **缓存预热**：系统启动时加载热点数据

### 8.3 消息队列优化

- **分区策略**：合理设置分区数
- **消费者组**：使用消费者组提高消费能力
- **消息重试**：实现消息重试机制

## 9. 总结

本数据库设计方案基于项目需求和架构方案，设计了MySQL关系型数据库和MongoDB文档数据库的表结构，以及Redis缓存和Kafka消息队列的使用方案。通过合理的数据库设计和优化，可以提高系统的性能、可靠性和可扩展性，满足项目的业务需求。
